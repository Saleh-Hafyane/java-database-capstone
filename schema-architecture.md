User Interaction & Entry: The process begins at the Presentation Tier where a user interacts with the system. This is done either via a browser-based Thymeleaf Dashboard (for Admins and Doctors) or through a REST Client (for patient records and appointments).
Request Routing: The request is sent to the Controller Layer. The application determines if the request requires a server-rendered view (handled by Thymeleaf Controllers) or raw data exchange (handled by REST Controllers).
Service Layer Delegation: The controller passes the request to the Service Layer. This central hub processes the business logic, such as validating user permissions or calculating appointment slots, ensuring the controllers remain "thin."
Data Access Coordination: The Service Layer identifies which data source is needed and communicates with the Repository Layer. It directs traffic to either the JPA Repositories for relational data or MongoDB Repositories for document-based data.
Database Execution: The repositories execute queries against the Data Tier. MySQL manages structured entities like Patient and Doctor profiles, while MongoDB handles the flexible, unstructured data found in medical Prescriptions.
Model Binding & Transformation: Once the database returns the data, it is mapped into Java objects. Relational rows become JPA Entities, and document entries become POJOs (Plain Old Java Objects), providing a unified data format for the backend logic.
Response Finalization: The processed data is sent back through the controller to the user. In the MVC flow, the data is injected into a Thymeleaf template to generate HTML; in the REST flow, the data is serialized into JSON format for the client.
